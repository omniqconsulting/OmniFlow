Scope of This Brief 1.1 What This Brief Covers This brief covers two things that must be built together as a single unit: (1) the WhatsApp sending foundation — shared infrastructure every future template will reuse — and (2) the first live pipeline, omniflow_ticket_assigned, wired end-to-end and ready to test against a real phone number. MSG91 service module — the single point of contact with the MSG91 API Manual mobile number validation — &quot;Mark as Validated&quot; control, replacing OTP entirely WhatsApp send-attempt logging — one shared table, reused by all 13 templates over time Send-status visibility on the ticket detail timeline, with Resend shown only on failure The ticket_assigned trigger wired to fire a real WhatsApp message 1.2 What This Brief Explicitly Supersedes The OTP &amp; MSG91 Development Amendment v1.1 (found in the existing codebase) planned a full OTP verification system — account-creation OTP, login OTP, a 24-hour trusted session window, otp_tokens table, and new auth middleware checks. That entire approach is dropped. Per direct decision: phone number &quot;validation&quot; is now a manual one-click action performed by an Admin (or Super Admin), not a system-verified OTP flow. SUPERSEDED Sections 2, 4.1–4.4, and 6.1–6.3 of the OTP &amp; MSG91 Amendment v1.1 (schema, login flow changes, OTP pages) no longer apply. Do not implement them. Section 3 (MSG91 service module pattern) and Section 8 (security rules) are conceptually reused but rebuilt below to match the simpler, OTP-free model. 1.3 What This Brief Does NOT Cover (future briefs) The other 12 approved templates (checklist_due, checklist_overdue, ticket_escalated, ticket_unacknowledged, fms_stage_transition, registration_received, registration_rejected, flow_request_alert_sa / _approved / _rejected / _clarification, flow_deployed) — each gets its own brief, built and tested one at a time Inventory/material templates — out of scope for this solution entirely OTP of any kind — dropped permanently, not deferred SMS fallback — WhatsApp only, per direct decision The extended ticket_assigned variant with an appended task list — parked; requires a new Meta template submission and is not part of this brief 2. Key Decisions Locked Decision Outcome Phone validation model Manual &quot;Mark as Validated&quot; toggle, set by Admin/SA. Not OTP. Does not gate login — gates WhatsApp eligibility only. Delivery channel WhatsApp only. No SMS fallback for this pipeline. Failure visibility A &quot;Resend&quot; action is shown ONLY when a send has status FAILED. Successful sends show a quiet confirmation line, no button. Skipped (unverified) sends show a quiet skip line, no button. Build sequencing One template at a time. Foundation is built once now; each subsequent brief adds exactly one more wired trigger. Resilience rule A WhatsApp send failure must never block or roll back the underlying action (e.g. ticket assignment still succeeds even if the WhatsApp call fails). Template source of truth The MSG91-exported JSON per template (msg91_template_id + exact approved variable order) — not assumptions, not prose descriptions. 3. Follow the existing String-primary-key / new_id() / Text-for-detail convention already used by FMSEvent and TicketEvent in app/database.py — do not introduce a native JSON column type (not currently used anywhere in this codebase). class WhatsAppMessageLog(Base): &quot;&quot;&quot; Foundation table — every outbound WhatsApp send attempt across all pipelines is logged here, success or failure. Built once; reused by every template brief going forward. &quot;&quot;&quot; __tablename__ = &quot;whatsapp_message_log&quot; id = Column(String, primary_key=True, default=new_id) tenant_id = Column(String, ForeignKey(&quot;tenants.id&quot;), nullable=False) template_name = Column(String, nullable=False) recipient_user_id = Column(String, ForeignKey(&quot;users.id&quot;), nullable=True) recipient_phone = Column(String, nullable=False) variables_json = Column(Text, nullable=False) # JSON list, approved order status = Column(String, nullable=False) # SENT / FAILED / SKIPPED_UNVERIFIED error_message = Column(Text, nullable=True) related_entity_type = Column(String, nullable=True) # &apos;ticket&apos; for this brief related_entity_id = Column(String, nullable=True) attempt_count = Column(Integer, default=1) created_at = Column(DateTime, default=datetime.utcnow) last_attempted_at = Column(DateTime, default=datetime.utcnow) tenant = relationship(&quot;Tenant&quot;) recipient = relationship(&quot;User&quot;, foreign_keys=[recipient_user_id]) Add a viewonly convenience relationship on Ticket, mirroring the existing media relationship pattern already on that model: whatsapp_logs = relationship( &quot;WhatsAppMessageLog&quot;, primaryjoin=&quot;and_(WhatsAppMessageLog.related_entity_type==&apos;ticket&apos;, &quot; &quot;foreign(WhatsAppMessageLog.related_entity_id)==Ticket.id)&quot;, viewonly=True, order_by=&quot;WhatsAppMessageLog.created_at&quot;, ) NOTE variables_json stores the exact ordered list of values sent, so a later Resend can replay the identical send without recomputing ticket state that may have changed since (e.g. priority edited after the original failed send). This is why a dedicated log table is used instead of reusing the existing TicketEvent table, which has no field for replayable parameters and requires a non-null actor_id that does not fit a system-generated send. 4. MSG91 Service Module (Foundation) Single new file. No other file may call the MSG91 API directly — every future pipeline brief calls this same function. DESIGN NOTE This is a SYNCHRONOUS function (httpx.Client, not AsyncClient) by deliberate choice. notify_ticket_assigned() and the other notify_*() helpers in app/notifications.py are called from a mix of sync and async route handlers in app/main.py (e.g. move_ticket is sync, create_ticket is async). Keeping the send call synchronous avoids cascading async/await through every call site across the codebase — consistent with how broadcast_sync() already solves the identical problem for WebSocket events in the same file. NEW FILE — app/services/__init__.py # empty — marks app/services as a package NEW FILE — app/services/msg91.py &quot;&quot;&quot; app/services/msg91.py Single integration point for all outbound MSG91 WhatsApp calls. No other file should call the MSG91 API directly. Foundation module — built once, reused by every WhatsApp pipeline. &quot;&quot;&quot; import httpx import logging from app.constants import WHATSAPP_TEMPLATES, MSG91_AUTH_KEY, MSG91_WA_NUMBER logger = logging.getLogger(&quot;msg91&quot;) MSG91_BASE = &quot;https://control.msg91.com/api/v5&quot; def send_whatsapp_template(mobile: str, template_name: str, variables: list): &quot;&quot;&quot; Send a single WhatsApp message using a pre-approved Meta template. mobile: international format, e.g. &apos;919876543210&apos; — no &apos;+&apos;, no spaces template_name: key into WHATSAPP_TEMPLATES (app/constants.py) variables: ordered list matching the template&apos;s approved variable order exactly (see WHATSAPP_TEMPLATES for documented order) Returns (success: bool, error_message: str | None). NEVER raises — every failure path returns (False, reason). This function must never be allowed to break the action that triggered it. &quot;&quot;&quot; template = WHATSAPP_TEMPLATES.get(template_name) if not template: return False, f&quot;Unknown template: {template_name}&quot; if len(variables) != len(template[&quot;variable_order&quot;]): return False, ( f&quot;Variable count mismatch for {template_name}: &quot; f&quot;expected {len(template[&apos;variable_order&apos;])}, got {len(variables)}&quot; ) payload = { &quot;integrated_number&quot;: MSG91_WA_NUMBER, &quot;content_type&quot;: &quot;template&quot;, &quot;payload&quot;: { &quot;to&quot;: mobile, &quot;type&quot;: &quot;template&quot;, &quot;template&quot;: { &quot;name&quot;: template_name, &quot;language&quot;: {&quot;code&quot;: &quot;en&quot;}, &quot;components&quot;: [{ &quot;type&quot;: &quot;body&quot;, &quot;parameters&quot;: [{&quot;type&quot;: &quot;text&quot;, &quot;text&quot;: str(v)} for v in variables], }], }, }, } try: with httpx.Client(timeout=15.0) as client: resp = client.post( f&quot;{MSG91_BASE}/whatsapp/whatsapp-outbound-message/bulk/&quot;, headers={&quot;authkey&quot;: MSG91_AUTH_KEY, &quot;Content-Type&quot;: &quot;application/json&quot;}, json=payload, ) if resp.status_code == 200: return True, None return False, f&quot;MSG91 returned {resp.status_code}: {resp.text[:300]}&quot; except httpx.TimeoutException: return False, &quot;MSG91 request timed out&quot; except Exception as exc: logger.exception(&quot;MSG91 send failed for template=%s&quot;, template_name) return False, str(exc) def _ordinal(n: int) -&gt; str: if 11 &lt;= n % 100 &lt;= 13: return f&quot;{n}th&quot; return f&quot;{n}{ {1:&apos;st&apos;,2:&apos;nd&apos;,3:&apos;rd&apos;}.get(n % 10, &apos;th&apos;) }&quot; def format_wa_date(dt) -&gt; str: &quot;&quot;&quot; Matches the exact format approved in the omniflow_ticket_assigned Meta template example: &apos;24th Oct 2026&apos;. Use this anywhere a date is sent as a WhatsApp template variable, not strftime ad hoc, so every template stays consistent with what Meta approved. &quot;&quot;&quot; return f&quot;{_ordinal(dt.day)} {dt.strftime(&apos;%b %Y&apos;)}&quot; 5. Pipeline Wiring — omniflow_ticket_assigned notify_ticket_assigned() in app/notifications.py is called from three places already (ticket creation, bulk upload, and reassignment) — wiring the WhatsApp send inside this one shared function means all three call sites get WhatsApp behaviour automatically, with zero changes to app/main.py. CRITICAL The WhatsApp send must be wrapped so an exception anywhere in this path can never propagate back up into notify_ticket_assigned() and break ticket assignment.
