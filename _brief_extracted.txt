1. Executive Summary & Scope
This brief covers three connected pieces of work that together replace the current single-number MSG91 WhatsApp integration with a per-tenant Gupshup integration, and close the consent/evidence gap that the earlier manual verification model left open.
1.1  The three pillars
Per-tenant WABA configuration — every client (tenant) gets its own Gupshup app, WhatsApp number, and API credentials, instead of all tenants sharing one WABA. This isolates quality-rating and blocking risk per client.
QR-code opt-in with granular status — replaces the manual admin “mark verified” button as the primary consent mechanism. An employee scans their tenant's QR code, messages in on WhatsApp, and that inbound message becomes the proof of both a working phone number and genuine consent.
Super Admin consent & compliance log — every consent event and every outbound message is logged per tenant, with a one-click “Generate Appeal Package” export that produces evidence Sahil can submit to Meta/Gupshup if a client's number is ever flagged or blocked.
1.2  What this brief does NOT cover
Automated Gupshup Partner API onboarding of new tenants (create-app / embedded-signup-link generation called from OmniFlow itself). For this phase, WABA credentials are obtained manually (Section 3) and entered into the Super Admin Portal by Sahil. Full API-driven onboarding is a candidate for a future phase.
Automated follow-up nudges for employees stuck in PENDING status. Confirmed as a manual, human-driven decision — no APScheduler reminder job in this brief.
Migration of the 13 existing MSG91 templates — template re-submission for Meta approval under the new Gupshup WABA per tenant is a parallel manual/content task, not a code task, tracked separately per tenant during onboarding.
Any change to FMS, PMS, Dispatch, Invoicing, or other non-WhatsApp modules.

13
Template ID mapping
The whatsAppTemplateId field in send requests takes the Facebook temp ID (numeric), not the Gupshup temp ID (UUID) — confirmed from the live Templates page and the API sample side by side.
14
Webhook approach
A new, separate webhook is added per tenant in Gupshup's Channels → WhatsApp → Webhooks screen, pointed at OmniFlow's own endpoint. Pre-existing webhooks on a tenant's account (e.g. any Gupshup-product-owned webhook such as a CX/commerce integration) are never modified or removed — they belong to other Gupshup features and must be left exactly as configured.
15
Webhook payload format
Gupshup format (v2) is selected for OmniFlow's webhook — the normalized {app, type, payload, sender} shape — not Meta format (v3), since v2 is the documented and verified shape this brief is built against.
16
Webhook events subscribed
Message, Sent, Delivered, Read, Failed, Others. Not Enqueued, Delete, Billing, Payments, Flow Message, Template events, or System Events — out of scope for this brief.
17
Webhook authentication
A per-tenant random shared secret is set as a custom header (X-OmniFlow-Webhook-Secret) on the Gupshup webhook config. OmniFlow's endpoint rejects any inbound call whose header value doesn't match the stored secret for that tenant.
18
Webhook URL scheme
Per-tenant callback URL uses an opaque random token, not the raw tenant_id, to avoid an enumerable/guessable public endpoint: https://omniflow.omniqconsulting.com/webhooks/gupshup/{webhook_token}.

3. Prerequisites — Manual Setup Steps (Sahil, outside of code)
IMPORTANT
Nothing in this section is a Claude Code task. These are steps Sahil performs in the Gupshup Partner Portal and Meta Business Manager before, and alongside, each tenant's onboarding. Claude Code should assume these values already exist and simply provide the Super Admin Portal fields to store them (Section 4.1 / 7).

3.1  One-time platform setup
Sign up for a Gupshup Partner Portal account (not a self-serve single-app account) — this is what allows creating a separate app/WABA per client under one umbrella.
Complete Meta's Tech Provider Program (TPP) enrolment through Gupshup to obtain an approved Solution ID. This is required to onboard client WABAs at scale rather than one at a time through the console UI.
Set up a Partner Wallet in the Partner Portal (choose currency, e.g. USD) — this is the wallet that funds messaging usage across all client apps created under the partner account.
Configure a default callback/webhook URL for the partner account — this is where inbound messages, opt-in/opt-out events, and delivery-status events for ALL client apps will arrive by default (per-app override is possible later but not required for this brief).
3.2  Per-client (per-tenant) onboarding — repeat for every new client
In the Partner Portal, click ‘Create App’ to register a new Gupshup app for this client.
Choose Embedded Signup as the onboarding method (Meta-recommended). The client (or Sahil, acting on their behalf if the client can't complete it themselves) logs into the relevant Facebook Business Manager account, selects or creates a WhatsApp Business Account, and verifies a dedicated phone number for that client's WhatsApp Business number.
Complete Meta Business Verification for that client's business entity (GST + Certificate of Incorporation, same requirement already hit for OTP/authentication templates) — required before the number can send at full capability.
Once approved, Gupshup emails confirmation that the app/WABA is live. Note down, from the Integrations → APIs → WhatsApp API screen for this tenant's account: the Client ID, the Secret Token, and the registered source phone number (E.164 format).
Re-submit all relevant WhatsApp message templates for this client's new WABA and get them approved by Meta (existing template content can be reused, but template IDs are provider- and WABA-specific and must be recreated). Confirm both the Facebook temp ID and Gupshup temp ID are visible for each on the Templates → WhatsApp → Your Templates screen — the Facebook temp ID is what gets used as whatsAppTemplateId in send calls.
WHAT COMES NEXT
Once these four items are in hand (Client ID, Secret Token, phone number, approved templates), the rest of onboarding — configuring OmniFlow and finishing the Gupshup webhook — happens entirely in the Super Admin Portal. See Section 3.3, the day-to-day runbook for this part, once this brief has shipped.

WHY MANUAL FOR NOW (SECTION 3.2 ONLY)
Automating the Embedded Signup / WABA creation steps above end-to-end (calling Gupshup's Partner onboarding APIs directly from OmniFlow) is possible but adds meaningful scope — it's a good Phase 2 candidate once the Section 3.3 flow below is proven across a few real clients.

5. QR Code / Opt-In Link Generation & Display
5.1  What gets generated, and when
When a Super Admin enters a tenant's gupshup_source_number in the new WhatsApp Configuration form (Section 7), the system generates a wa.me deep link in the form https://wa.me/<source_number>?text=<prefilled opt-in message>, and stores it in tenants.whatsapp_opt_in_link.
A QR code image encoding that link is generated server-side (a lightweight QR-generation library is sufficient — no external API call needed) and displayed wherever the link is shown. It does not need to be stored as a file; render on request from the stored link.
The prefilled message text should be a short, clear opt-in phrase, e.g. “Hi, I'd like to receive WhatsApp updates from [Tenant Label: Employee/Worker] via OmniFlow” — pull the entity label from the tenant's existing label configuration system rather than hardcoding “Employee.”
5.2  Where it's displayed
Super Admin Portal → Client Detail → WhatsApp Configuration section: shows the QR code and the raw link, so Sahil can hand it to the client directly (e.g. via WhatsApp or email) during onboarding.
Tenant-side Admin Employees tab: a small “Share opt-in QR” action/panel so the tenant's own Admin can show it to new employees directly, without needing to go through Sahil each time.
5.3  What is explicitly NOT in scope here
Printing/export to physical collateral (posters, ID cards) — the raw QR image render is sufficient; any physical-format export is a future nice-to-have.

6. Webhook Handler Updates
6.0  Endpoint, routing, and authentication (new in v1.1)
New route: POST /webhooks/gupshup/{webhook_token} — one path per tenant, where {webhook_token} is the opaque value stored in tenants.gupshup_webhook_token (Section 4.1), not the raw tenant_id.
On receipt, look up the tenant by gupshup_webhook_token from the URL path. No match: return 404 and log the attempt — do not process the payload.
Verify the X-OmniFlow-Webhook-Secret request header against that tenant's stored gupshup_webhook_secret using a constant-time comparison. Mismatch or missing header: return 401 and do not process the payload.
Only after both checks pass, parse the body as Gupshup format (v2) — the {app, timestamp, version, type, payload} shape documented in Gupshup's Inbound Messages and Events reference — and route by the type field (message / user-event / message-event, etc.) to the handling logic in 6.1–6.3 below.
WHY A DEDICATED ROUTE PER TENANT
This is what keeps future tenant onboarding a Super Admin Portal data-entry task rather than a code change: Sahil generates a token and secret for a new tenant in the SA portal, pastes the resulting URL and secret into that tenant's new webhook in Gupshup's console, and the route above already handles it — no new endpoint, no deployment.

6.1  Inbound message webhook — opt-in matching
The tenant is already known from the URL path (6.0) — no need to derive it from the app/source number in the payload.
Normalize the sender's phone number to the same E.164 format used when employees are added (lock this format now — e.g. always store and compare as +<countrycode><number> with no spaces/dashes — to avoid silent match failures from formatting drift).
Look up an employee within that tenant whose on-file phone number matches the normalized sender number.
Match found, employee currently PENDING or MISMATCH: set whatsapp_opt_in_status = OPTED_IN, opt_in_source = QR, opt_in_at = now, matched_phone = sender number. Insert a whatsapp_consent_events row, event_type = OPT_IN_RECEIVED, with the Gupshup message ID and full raw payload.
Match found, employee already OPTED_IN or MANUALLY_VERIFIED: still insert the consent_events row (useful trail of repeated engagement) but do not change status.
No match found: still insert a whatsapp_consent_events row with employee_id = null, so there's a record of unmatched inbound activity on that tenant's number for troubleshooting — do not silently drop it.
6.2  Opt-out handling
Gupshup's inbound event stream emits opted-out user-events (and/or a STOP keyword reply, depending on what the webhook payload actually contains — verify against a live payload during testing). On receipt, set whatsapp_opt_in_status = OPTED_OUT for the matched employee and insert an OPT_OUT_RECEIVED consent event.
An OPTED_OUT employee must not receive further operational WhatsApp sends until manually re-validated or until they opt in again via QR.
6.3  Outbound status webhook — delivery history
On receiving a message-status event (enqueued / sent / delivered / read / failed) for a message this tenant sent, find the corresponding whatsapp_logs row by messageId, append a {status, timestamp} entry to delivery_status_history, and append the raw payload to raw_status_webhook_payloads — do not overwrite prior entries.
6.4  Resilience rule (carried over from existing pattern)
NON-BLOCKING
A webhook processing failure (bad payload, DB write error) must never crash the receiving endpoint or block Gupshup's retry mechanism. Log and continue, per the existing OmniFlow rule that notification-layer failures must never break the underlying action.

How to trigger
Expected result
